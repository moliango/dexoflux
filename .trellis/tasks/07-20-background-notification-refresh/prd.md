# PRD：后台通知刷新

## 目标

在 DexoFlux 进入后台后，由 iOS 在系统允许的时间窗口唤醒应用，检查已登录论坛的新通知并投递本地通知，同时保持 App 图标角标与未读状态一致。

## 已确认事实

- 当前通知协调器只在前台每 60 秒轮询，进入后台后会停止计时器。
- FluxDo 使用 iOS `BGTaskScheduler` 定期唤醒并执行一次短网络请求，不是 APNs 实时推送。
- iOS 后台刷新最早可设置为 15 分钟，但实际触发时间完全由系统调度，不能承诺准点或持续运行。
- DexoFlux 支持多个论坛和账号，不能照搬 FluxDo 只保存一个 `userId` 的后台状态。
- 现有通知游标已经按 `baseURL + username` 隔离，并能避免首次同步轰炸历史通知。

## 子任务

1. `07-20-background-refresh-runtime`：注册、调度和完成 iOS `BGAppRefreshTask`。
2. `07-20-background-notification-sync`：无界面网络同步核心、后台 API 上下文和多论坛结果。
3. `07-20-background-notification-delivery`：共享游标、持久角标、通知权限和端到端集成验证。

## 需求

1. App 启动时注册后台刷新任务，进入后台时按需提交下一次任务。
2. 任务触发后立即续约下一次请求，并在系统过期时取消正在执行的网络工作。
3. 仅处理仍有有效 Discourse `_t` Cookie 的论坛。
4. 所有已登录论坛按账号隔离游标，首次后台同步只建立基线。
5. 后台发现的新通知复用现有本地通知样式和点击路由。
6. 后台同步后更新 App 图标总角标；没有已登录论坛时清除旧角标。
7. 单个论坛失败不阻止其他论坛执行，任务结果能够反映部分或全部失败。
8. 不在后台主动弹权限申请、Cloudflare 验证页或其他 UI。

## 验收标准

- [x] `Info.plist` 声明允许的任务标识符和 `fetch` 后台模式。
- [x] 后台任务在 `didFinishLaunchingWithOptions` 返回前完成注册。
- [x] 进入后台时只在存在可同步账号时提交 `BGAppRefreshTaskRequest`。
- [x] 任务执行时重新调度下一次任务，并确保 `setTaskCompleted` 只调用一次。
- [x] Expiration 会取消同步任务，取消不会被当作成功。
- [x] 多论坛同步复用相同通知游标，不重复或批量提醒历史通知。
- [x] 本地通知点击仍能进入正确论坛、话题和楼层。
- [x] 前台现有红点和 60 秒轮询行为不回归。
- [x] 新增策略测试、通知游标测试及 iOS 测试构建通过。

## 不在范围

- APNs 服务端推送和 App 被杀死后的实时保证。
- 通过音频、定位或持续后台网络规避 iOS 调度限制。
- 在本任务中新增用户可见的后台刷新开关。
